PUT THE RELEASE KEYSTORE IN THIS FOLDER (mobile/)
==================================================

WHAT
    The production signing key for the sideloaded Android APK. Every release
    must be signed with the SAME key -- Android refuses to install an update
    signed by a different one, and officers would have to uninstall and lose
    local state to recover.

    Conventional name and location (what this project's setup expects):

        mobile/firenet-release.keystore
        key alias: firenet

    The path is not actually hardcoded -- it is read from a Gradle property
    (see below), so any location works. Keeping it here is just the convention.

WHERE TO GET IT
    This project has already shipped releases, so the keystore already
    exists -- it lives in the production server's own mobile/ folder (the
    server holds a full clone at ~/www/firenet, the same layout as yours).
    Pull it from there rather than creating a new one:

        scp <your-ssh-user>@wildfire.forest.go.th:~/www/firenet/mobile/firenet-release.keystore \
            mobile/firenet-release.keystore

    The four Gradle properties (passwords/alias) are NOT on the server in any
    file -- they were never written down alongside the keystore on purpose.
    Ask a maintainer for those separately.

    It is deliberately NOT in git, and it CANNOT be regenerated -- a new
    keystore is a different identity and permanently breaks over-the-top
    updates for every already-installed device. For THIS project you should
    essentially never need "HOW TO CREATE A NEW ONE" below -- it exists for
    bootstrapping a *different, brand-new* deployment that has never shipped
    a release, not for this one.

    If you are only doing local development you do NOT need this file. Debug
    and dev-client builds sign themselves.

HOW TO CREATE A NEW ONE (first release of a brand-new deployment only)
    Not applicable to this project -- see above, the keystore already exists
    on the server. Keep this section only in case you're bootstrapping an
    unrelated deployment from this same codebase.

    Do this once, ever, per deployment -- never "just to fix a problem" on an
    existing one. If officers already have a build installed that was signed
    with a different key, a new keystore permanently orphans them (they must
    uninstall and lose local state). If you're not sure whether one already
    exists for the deployment you're working on, ask a maintainer before
    running this.

    Requires a JDK (the `keytool` command ships with it -- the same JDK you
    use to build the Android app already has it on PATH).

        keytool -genkeypair -v \
          -keystore firenet-release.keystore \
          -alias firenet \
          -keyalg RSA -keysize 2048 -validity 10000 \
          -dname "CN=FireNET, OU=wildfire, O=<your org>, L=<city>, ST=<state>, C=<country code>"

    keytool will prompt for a keystore password and a key password twice each
    (type them in -- they will not echo). Using the same value for both is
    fine and simplifies the Gradle properties below. `-validity 10000` is
    ~27 years, deliberately long: this key must outlive every device it has
    ever signed a build for, and renewing it later means starting over with a
    new identity.

    What you get is one file (`firenet-release.keystore`) holding both the
    private key and its self-signed certificate. Nothing here talks to
    Google, a Firebase project, or any external service -- it's a local
    cryptographic identity you're minting for this project, not registering
    anywhere.

    Immediately back up the resulting file AND both passwords somewhere
    durable and access-controlled (a password manager or the org's secrets
    store, not just this machine) -- see WARNING below for why losing it is
    unrecoverable. Then follow HOW TO INSTALL IT.

HOW TO INSTALL IT
    1. Put the .keystore file here (or anywhere you prefer).
    2. Add these four properties to your GLOBAL Gradle config, at
       ~/.gradle/gradle.properties -- NOT to any file inside this repo:

           FIRENET_UPLOAD_STORE_FILE=<absolute-path-to-your-clone>/mobile/firenet-release.keystore
           FIRENET_UPLOAD_STORE_PASSWORD=<ask a maintainer, or the one you set with keytool>
           FIRENET_UPLOAD_KEY_ALIAS=firenet
           FIRENET_UPLOAD_KEY_PASSWORD=<ask a maintainer, or the one you set with keytool>

       <absolute-path-to-your-clone> means YOUR machine's actual path to this
       repo -- e.g. if you cloned to C:\Projects\FireNET-RFD, the line reads
       FIRENET_UPLOAD_STORE_FILE=C:/Projects/FireNET-RFD/mobile/firenet-release.keystore.
       This is NOT a fixed convention string -- it is example-specific to
       whoever wrote it, and every clone location is different. Copying it
       verbatim points at a file that doesn't exist on your machine, and
       assembleRelease fails loudly at the signing step ("keystore file does
       not exist") -- confusing the first time, but at least it stops the
       build rather than shipping a bad APK. Use forward slashes in the path,
       even on Windows.

       Verify it actually took effect before trusting a release: build once,
       then run `apksigner verify --print-certs <apk>` and confirm the
       certificate is yours, not "Android Debug".

    These are consumed by mobile/plugins/withReleaseSigning.js, which rewrites
    the generated android/app/build.gradle at prebuild time.

WARNING -- THE FAILURE MODE HERE IS SILENT
    If those properties are missing, the release build does NOT fail. It falls
    back to the DEBUG keystore (see withReleaseSigning.js). You get an APK that
    looks fine, publishes fine, and then refuses to install over any previous
    real release -- signed by the wrong identity. publish-apk.ps1 verifies the
    version code but does NOT verify the signing key.

    To check which key an APK was actually signed with:

        apksigner verify --print-certs <path-to.apk>

    The certificate should match previous releases, not "Android Debug".

DO NOT COMMIT IT
    Already covered by mobile/.gitignore (`*.keystore`, `*.jks`,
    `keystore.properties`). Losing or leaking this key is unrecoverable.

LEAVE THIS .TXT IN PLACE
    It is tracked in git as a marker for the next person setting up. Deleting it
    locally just shows up as a pending deletion in your working tree.
